Multi-Room Hidden-Camera Detection Protocol

This protocol guides a user through detecting motion-activated Wi-Fi monitoring devices in a house or multi-room apartment. It coordinates room actions with Wi-Fi traffic capture so researchers can look for devices whose network activity repeatedly responds to room activity.

Overview

The app instructs the user to enter, walk inside, and exit rooms at scheduled times while a Wi-Fi sniffer records surrounding traffic. For each room, the app records a binary action signal that marks when the user is actively testing that room.

After the run, researchers compare the room action signals with device network traffic. A device is a candidate if its traffic repeatedly changes during the same room actions and stays quiet during idle periods. This protocol does not prove that a device is a camera, and it should not be presented as a safety guarantee.

Privacy caution

Use this protocol only for controlled studies or user-guided privacy checks. Do not publish sensitive layouts, identities, addresses, or device identifiers.

Roles

Researcher/operator

Sets up the study, configures rooms, places QR codes or NFC tags if used, starts network capture, reviews logs, and handles consent and privacy.

User/participant

Follows app instructions, performs natural room actions, scans, taps, or presses buttons as instructed, and reports mistakes when something goes wrong.

Key Idea: One Action Signal Per Room

For a home with N rooms, the app records N action signals: 

A(t) = [a_1(t), a_2(t), ..., a_N(t)]

For each room R_i:

  • a_i(t) = 1 when the user is performing the detection action in room R_i.
  • a_i(t) = 0 otherwise.

During normal operation, at most one room signal should be 1 at a time. During idle periods, all room signals are 0.

Detection Methods Compared

The three conditions differ only in how the app records the room action interval. The detection algorithm, room order, action schedule, and sniffer setup should stay constant across methods.

Comparison of action-signal recording methods
Method Setup assumption Participant action Recorded interval
A. AR + click button The app assumes a room map is already available on the phone. It knows room boundaries or room labels. The user taps Start Action, enters the instructed room, walks naturally, exits, and taps Action Finished. The target room is active between the two button events. AR tracking can help verify movement and room entry or exit, but the core interval is based on user interaction and app state.
B. QR code The app does not assume a room map. The QR code IDs are already known to the app, and the room-to-code mapping is created during setup. The user matches each room with a QR code, then scans that room's code before entering and scans the same code again after exiting. The mapped room is active between the two scans.
C. NFC tag The app does not assume a room map. The NFC tag IDs are already known to the app, and the room-to-tag mapping is created during setup. The user matches each room with an NFC tag, then taps that room's tag before entering and taps the same tag again after exiting. The mapped room is active between the two taps.

QR/NFC Setup Phase

QR and NFC conditions do not need a geometric room map. They only need a mapping: 

room name <-> tag ID
  1. User enters the number of rooms to scan.
  2. For each room, the app asks the user to go to the room entrance.
  3. User scans a QR code or taps an NFC tag.
  4. User enters a room name or selects a room type.
  5. App stores the mapping.
  6. App shows a verification table.
  7. Detection starts only after the user confirms all mappings.

Detection Phases

QR/NFC detection

  1. The app tells the user which room to test.
  2. User stands outside that room.
  3. User scans or taps that room's QR/NFC tag before entering.
  4. User enters, walks naturally, and exits.
  5. User scans or taps the same tag again after exiting.
  6. The app records the interval between the two scans or taps as action=1 for that room.
  7. All other room action signals remain 0.

AR + button detection

  1. The app already has the room map.
  2. The app tells the user which room to test.
  3. User stands outside the target room.
  4. User taps Start Action.
  5. User enters, walks naturally, and exits.
  6. User taps Action Finished after exiting.
  7. The app records that interval as action=1 for the target room.
  8. All other room action signals remain 0.

Researcher note

Keep prompts short during the run. If the participant makes a mistake, log it, restart that action step, and preserve the original event record for analysis.

Error Handling and Validation

  • Only one room action can be active at a time.
  • For QR/NFC, the same tag must start and end the action.
  • If the app asks for Bedroom but the user scans Kitchen, the app should show an error.
  • If an action is active for Bedroom and the user scans a different room, the app should reject it or ask the user to report a mistake.
  • Add a Report Mistake button.
  • Mistakes include wrong room, wrong scan or tap, forgotten scan or tap, failed scan, failed NFC tap, entering too early, or exiting late.
  • The app should log mistakes and restart that action step.

Multi-Room Scheduling

Sequential mode

  • Test one room at a time.
  • Keep the room idle for the reset or cooldown period.
  • Perform four room actions separated by idle intervals.
  • Run detection.
  • Move to the next room.
  • This is slower but cleaner.

Interleaved mode

  • Use idle time for one room to test another room.
  • Never schedule overlapping actions.
  • Never enter a room during its required idle period.
  • Keep at least a short separation between actions in different rooms.
  • Avoid identical periodic schedules across rooms.
  • This is faster but requires careful logging.

Room state machine

Room state ∈ {SETUP, RESETTING, READY_FOR_ACTION, ACTION_ACTIVE, WAITING_IDLE, CLASSIFIED, NEEDS_DISAMBIGUATION}

Disambiguation

A device may correlate with multiple rooms if it sees a hallway, doorway, or more than one room. Use extra actions to separate room-specific signals from shared-view signals.

  1. Select candidate rooms one at a time.
  2. Keep other candidate rooms idle.
  3. Perform extra actions in only one room.
  4. Check whether the same device responds.
  5. Label the result as room-specific monitoring device, shared-view or hallway monitoring device, or unresolved candidate.

Data to Log

Use random participant and home identifiers. Keep consent records separate from technical logs.

Category Fields

Metrics

Detection metrics

  • Room-level true positive, false positive, and false negative
  • Device-level true positive, false positive, and false negative
  • Time to decision
  • Number of actions required
  • Unresolved cases

Action-signal quality

  • Missing action events
  • Extra action events
  • Start-time error
  • End-time error
  • Room-label error
  • Timestamp alignment error

Usability

  • Setup time and completion time
  • Number of mistakes
  • Number of scan/tap failures
  • Number of times help was needed
  • Perceived ease and confidence
  • Preferred method

Practical deployment

  • Setup burden and room mapping burden
  • Lighting dependence for QR
  • Phone-positioning burden for NFC
  • AR drift or tracking loss
  • Hallway/shared-view ambiguity

Participant Instructions

AR condition

  1. Stand outside the room shown in the app.
  2. Tap Start Action.
  3. Enter the room naturally.
  4. Walk along the instructed path.
  5. Exit the room.
  6. Tap Action Finished.
  7. Wait for the next prompt.

QR condition

  1. During setup, match each room to one QR code.
  2. During detection, scan the room QR code before entering.
  3. Enter, walk naturally, and exit.
  4. Scan the same QR code again after exiting.
  5. Wait for the next prompt.

NFC condition

  1. During setup, match each room to one NFC tag.
  2. During detection, tap the room NFC tag before entering.
  3. Enter, walk naturally, and exit.
  4. Tap the same NFC tag again after exiting.
  5. Wait for the next prompt.

Privacy and Ethics

Privacy caution

  • This protocol is for controlled research studies and user-guided privacy checks.
  • Obtain consent and institutional approval when running human-subject studies.
  • Do not collect audio or video unless separately justified and consented.
  • Do not decrypt bystander traffic.
  • Pseudonymize MAC addresses.
  • Do not store exact home addresses in the public dataset.
  • Separate consent forms from technical logs.
  • Participants may stop at any time.
  • Do not publish sensitive layouts, identities, or device identifiers.
